# icoo_proxy 桌面端 UI 优化设计文档

版本：v0.1  
日期：2026-07-01  
范围：`icoo_desktop/frontend` 桌面管理台  
目标平台：Windows 桌面端，Wails + Vue

## 1. 产品定位

icoo_proxy 桌面端是一个本地 AI 协议转换网关控制台，核心用户是在本机长期运行、配置、观察和排障的开发者或高阶用户。

界面不应走营销型、展示型路线，而应保持高信息密度、低干扰、快速扫视、快速操作的桌面工具气质。现有风格已经接近 VS Code / Linear / Raycast 的紧凑控制台方向，后续优化应延续这一基础，而不是彻底换皮。

## 2. 当前 UI 现状

### 2.1 已具备的优势

- 整体结构稳定：标题栏、侧边导航、顶部操作区、主内容区、状态栏的桌面框架清晰。
- 视觉方向明确：中性灰背景、白色面板、1px 边框、较少阴影，适合控制台类产品。
- 组件体系初具规模：已有 `UButton`、`UTable`、`UInput`、`USelect`、`UModal`、`UTag`、`USwitch` 等基础组件。
- 关键操作集中：页面级主操作通过 Teleport 放到右上角，符合桌面应用习惯。
- 表格能力较完整：支持固定列、横向滚动、分页、查询区、空状态。
- 设置页已有主题色和尺寸偏好，说明产品具备可配置设计系统的基础。

### 2.2 主要体验问题

#### 问题 A：视觉层级偏平均，关键状态不够突出

概览页、流量页、供应商页都使用统计卡片 + 面板 / 表格布局，但卡片视觉权重接近，真正需要优先感知的状态没有形成明确排序。

例如：

- 概览页中“运行状态 / 监听地址 / 访问模式 / 供应商状态”应比接口路径更重要。
- 流量监控中“错误请求、平均耗时、最近请求”应形成警戒优先级。
- 供应商页中“健康检查异常 / 未检查”比“供应商总数”更需要被看见。

#### 问题 B：宽表格依赖横向滚动，阅读路径压力较大

多个页面设置了较大的 `min-width`，例如流量表约 1680px、供应商表约 1640px。桌面端可以接受横向滚动，但当前表格把“识别对象、状态、诊断、操作”拆得过宽，导致用户需要在横向滚动中拼接上下文。

典型影响：

- 流量页请求 ID、端点、请求信息、路由、模型、请求体、错误信息同时出现，扫视成本高。
- 供应商页协议、地址、UA、Key、模型、健康状态并列展示，右侧操作容易离上下文太远。
- 空数据时仍出现横向滚动条，视觉上像“缺内容但有可滚动区域”。

#### 问题 C：页面命名存在中英混用

侧边栏中“Provider”和“授权 Key”等命名混用，页面标题也出现“Provider”。这会削弱产品的统一感。

建议面向中文界面统一：

- Provider -> 供应商
- Key -> 密钥或授权 Key，根据页面语义保留“授权 Key”
- only_stream -> 仅流式
- max_tokens -> 最大 Token 数，必要时保留技术字段作为辅助说明

#### 问题 D：状态表达较分散

状态同时出现在标题栏、状态栏、统计卡片、表格标签中，但没有统一的“严重程度”表达。

当前状态色可用，但需要更明确的语义层级：

- 正常：绿色，仅用于真正可用或成功。
- 注意：橙色 / 黄色，用于未配置、未检查、警告、缺少密钥。
- 异常：红色，用于错误请求、不可达、保存失败。
- 中性：灰色，用于内置、默认、无数据。
- 信息：蓝色，用于协议、路径、可点击辅助状态。

#### 问题 E：表单编辑成本偏高

供应商、模型、端点、设置页的表单字段较技术化。字段本身正确，但缺少分组提示和默认值解释。

问题不是“看不懂”，而是用户每次编辑都要重新判断：

- 这个字段是否必填？
- 留空会怎样？
- 保存后是否需要重载？
- 当前改动是否立即生效？

#### 问题 F：空 / 加载 / 错误状态可用但不够行动化

当前已有空状态文字，例如“当前尚未配置供应商”。但空状态通常没有下一步操作入口，用户需要移动到右上角寻找“新建”。

建议空状态直接提供对应主操作：

- 无供应商：添加供应商
- 无授权 Key：新增授权 Key / 继续本地信任模式
- 无流量：刷新流量 / 检查代理地址
- 无端点：新增端点 / 查看内置端点说明

## 3. 优化目标

### 3.1 体验目标

1. 首屏 3 秒内看懂网关是否可用。
2. 主要配置页 5 秒内定位异常项。
3. 表格横向滚动减少 30% 以上，关键列不再依赖滚动才能理解。
4. 新建 / 编辑表单降低技术字段理解成本。
5. 所有页面的空、加载、错误状态都有明确下一步。

### 3.2 视觉目标

- 保持紧凑桌面控制台风格。
- 降低大面积浅蓝背景的存在感，让内容面板更干净。
- 用状态色表达业务含义，而不是装饰色。
- 控件继续保持 26-34px 高度，避免 Web 营销页式大间距。
- 卡片圆角维持 6px 以内，边框优先于阴影。

## 4. 设计方向

推荐方向：**Neutral Command Console**

关键词：

- 紧凑
- 中性
- 数据优先
- 状态清晰
- 低装饰
- 操作就近

参考气质：

- VS Code 的工具面板结构
- Linear 的轻量状态表达
- Raycast 的命令式简洁
- 数据库客户端 / API 客户端的高密度表格

## 5. 信息架构优化

### 5.1 侧边栏导航

现有分组：

- 网关
- 配置
- UED

建议调整为：

- 运行
  - 网关概览
  - 流量监控
- 配置
  - 供应商
  - 路由规则
  - 模型路由
  - 端点
  - 授权 Key
- 系统
  - 项目设置
  - 组件规范

优化点：

- 将“项目设置”从配置中拆到系统，避免和业务配置混杂。
- 将“Provider”改为“供应商”。
- 侧边栏折叠状态下保留 tooltip，展开状态下不显示空 description 行。
- 当前路由高亮左侧竖线可保留，但 active 背景应更轻，避免导航抢过内容主视觉。

### 5.2 顶部栏

当前顶部栏只有面包屑和页面操作。建议补充页面级标题，但保持紧凑：

结构：

- 左侧：页面标题 + 低权重路径
- 右侧：主操作 + 次操作 + 危险操作

示例：

- 标题：流量监控
- 辅助：icoo_proxy / 运行 / 流量监控
- 操作：刷新、自动刷新、清空

操作排序规则：

1. 主操作放最右或最醒目。
2. 危险操作不应紧邻主要确认按钮，可增加分隔或置为次级红色描边。
3. 开关类操作与刷新类操作分组，避免“清空请求”和“自动刷新”视觉上并列。

## 6. 关键页面优化方案

### 6.1 网关概览

目标：成为“是否正常”的第一判断页面。

建议布局：

1. 顶部状态带
   - 服务状态：运行中 / 未连接 / 异常
   - 监听地址
   - 授权模式
   - 最近错误

2. 核心指标卡
   - 供应商可用数
   - 已启用路由
   - 授权 Key 数
   - 最近请求量

3. 诊断面板
   - 上游就绪状态
   - 供应商健康汇总
   - 接口路径列表

现有问题修复：

- “支持的接口路径”不要和“上游就绪状态”同权重。它是参考信息，应放在次级区域。
- “供应商健康汇总”应提前，尤其是异常 / 未检查数量。
- 概览页首屏尽量减少空白，卡片高度统一。

### 6.2 流量监控

目标：快速定位失败请求、慢请求、路由命中问题。

建议布局：

1. 统计区改为两层
   - 第一层：最近请求、错误请求、平均耗时、总 Token
   - 第二层：成功请求、输入 Token、输出 Token

2. 表格默认列重新组织
   - 固定左列：时间 + 状态 + 方法 / 端点
   - 主体列：模型路由、耗时、Token、错误摘要
   - 详情信息进入行展开或详情抽屉

3. 增加筛选
   - 协议
   - 状态：全部 / 成功 / 错误
   - 耗时：全部 / 慢请求
   - 是否有错误

4. 行详情
   - 请求信息
   - User-Agent
   - 请求体预览
   - 路由命中
   - 错误堆栈或错误详情

降低横向滚动的关键策略：

- 表格主视图只保留排障必需字段。
- 长文本字段用详情抽屉承载。
- 状态码与耗时合并到同一诊断列。

### 6.3 供应商

目标：让用户知道哪些上游可用、哪些需要处理。

建议布局：

1. 顶部统计卡改为：
   - 总数
   - 可用
   - 警告 / 异常
   - 未检查

2. 表格列优化：
   - 供应商：名称、协议、描述
   - 连接：Base URL、API Key 状态、User-Agent
   - 模型：模型数量、主模型
   - 健康：状态、耗时、HTTP 状态
   - 状态 / 操作

3. 操作区就近：
   - 检查
   - 编辑
   - 模型
   - 删除

4. 表单优化：
   - 分组为“基础信息”“连接设置”“模型获取”“行为开关”
   - API Key 字段说明：编辑时留空将保留已有密钥
   - only_stream 文案改为“仅使用流式请求”

### 6.4 路由规则

目标：清晰表达“下游协议 -> 供应商 -> 上游协议”的映射关系。

建议保留当前映射表思路，但视觉上做成规则矩阵：

- 左：下游协议
- 中：当前供应商
- 右：上游协议与状态
- 操作：配置 / 修改

优化点：

- “未配置”状态应比普通文字更醒目。
- 无供应商可选时，在表格上方出现提示并引导去新增供应商。
- 强制切换确认弹窗需要强调“只影响后续请求”。

### 6.5 模型路由

目标：让别名到真实模型的关系更容易扫视。

建议：

- 别名列显示：别名 + 启用状态
- 目标列显示：供应商 / 协议 / 模型
- 更新时间放低权重，不与主信息并列。
- 新建表单中，选择供应商后再显示“获取模型”按钮，减少初始干扰。

### 6.6 端点

目标：把内置端点、自定义端点、启用状态表达清楚。

建议：

- 路径列继续使用 code tag。
- 协议和类型合并：协议 tag + 内置 / 自定义 tag。
- 更新时间放入说明列的次级文本。
- 内置端点的删除按钮应直接隐藏或显示禁用原因 tooltip。

### 6.7 授权 Key

目标：帮助用户判断是否需要配置鉴权。

建议：

- 空状态分为两种：
  - 本地信任模式：提示“当前无需授权 Key”
  - 鉴权模式：提示“需要新增授权 Key 才能访问”
- 使用方式卡片可展示 Bearer 和 x-api-key 两种方式。
- Key 列默认只显示掩码，复制操作使用图标按钮。

### 6.8 项目设置

目标：降低环境变量式字段的理解成本。

建议：

- 表单分组保持：
  - 外观设置
  - 运行监听
  - 超时与限制
  - 日志记录
- 字段标签显示中文名，技术字段名作为次级说明。
  - 代理主机 / `PROXY_HOST`
  - 代理端口 / `PROXY_PORT`
  - 默认最大 Token / `PROXY_DEFAULT_MAX_TOKENS`
- 对“保存并重载”增加保存成功后的状态反馈。
- 日志 body 开关应提示隐私和磁盘占用风险。

## 7. 设计系统补充

### 7.1 色彩 Token

保留当前主色可配置能力，补充语义规则：

```css
--ued-color-state-ok: #16a34a;
--ued-color-state-warn: #c2790a;
--ued-color-state-error: #dc2626;
--ued-color-state-info: #2563eb;
--ued-color-state-neutral: #6b7280;
```

建议降低页面背景蓝灰感：

```css
--ued-color-bg-page: #f5f6f8;
--ued-color-bg-shell: #eef0f3;
--ued-color-bg-card: #ffffff;
--ued-color-border: #d7dbe2;
```

### 7.2 字体与字号

继续使用当前字号体系：

- 页面标题：16-18px / 600
- 面板标题：14px / 650
- 正文：13px
- 辅助信息：12px
- 状态栏：11px

优化规则：

- 表格主信息使用 13px / 500。
- 表格辅助信息使用 12px / muted。
- 不使用负字距。
- 技术路径、模型名、Key、URL 使用等宽字体或 code tag。

### 7.3 间距与尺寸

建议继续：

- 页面边距：12px
- 面板内边距：12px
- 面板间距：12px
- 表格行高：34-42px
- 按钮高度：28px 默认
- 图标按钮：26-28px

需要统一：

- 统计卡高度建议固定 82-88px。
- 查询栏高度与表格头部间距统一。
- 空状态最小高度按页面类型区分，避免大面积空白。

### 7.4 组件规则

#### StatCard

新增两种权重：

- `priority="high"`：用于错误、运行状态、异常数量。
- `priority="normal"`：用于普通指标。

#### UTable

建议扩展：

- 行展开详情。
- 列密度模式：compact / normal。
- 空状态 action slot。
- 固定右侧操作列 hover 时保持背景一致。

#### UTag

统一文案：

- 启用 / 停用
- 已配置 / 未配置
- 可达 / 警告 / 不可达 / 未检查
- 内置 / 自定义
- 本地信任 / 需要鉴权

#### UModal

表单弹窗建议：

- 宽度 560px：简单配置。
- 宽度 640px：供应商等中型表单。
- 宽度 760px：模型批量编辑。
- footer 固定在底部，长表单时 body 滚动。

## 8. 交互与状态规范

### 8.1 加载状态

- 页面首次加载：内容区 skeleton 或紧凑 loading。
- 表格刷新：保留旧数据，表头显示刷新中，不清空表格。
- 按钮提交：按钮 loading，表单字段保持可读。

### 8.2 空状态

空状态必须包含：

- 当前状态说明。
- 为什么会出现。
- 下一步操作。

示例：

- “当前没有请求记录。可以刷新流量，或向 `http://127.0.0.1:18181` 发起一次请求。”
- “尚未配置供应商。添加供应商后才能配置路由规则。”

### 8.3 错误状态

- 全局错误显示在消息组件。
- 页面级错误应显示在内容顶部。
- 字段级错误显示在字段下方。
- 网络 / 服务未连接时，状态栏和页面顶部都应有提示。

### 8.4 危险操作

- 清空请求、删除供应商、删除端点、强制切换必须二次确认。
- 危险按钮默认不使用纯红实心，除确认弹窗中的最终确认外。
- 列表行内删除使用图标按钮 + tooltip + confirm。

## 9. 可访问性与键盘体验

- 所有图标按钮必须有 `aria-label` / `title`。
- 焦点环保持当前主色体系。
- 弹窗打开后焦点进入第一个可编辑字段，关闭后回到触发按钮。
- Escape 关闭非提交中弹窗。
- Enter 提交表单，危险确认不默认聚焦确认按钮。
- 状态色不能作为唯一信息来源，必须配合文字。

## 10. 实施计划

### Phase 1：低风险统一

- 统一导航中文命名。
- 调整顶部栏标题层级。
- 优化状态色语义和标签文案。
- 给空状态增加 action slot。
- 统一危险操作按钮风格。

### Phase 2：核心页面重构

- 流量监控表格列重组，增加行详情。
- 供应商表格分组显示，减少横向滚动。
- 概览页重新排序首屏状态模块。
- 项目设置字段改为中文主标签 + 技术字段辅助说明。

### Phase 3：组件能力增强

- `UTable` 增加行展开 / 详情抽屉能力。
- `StatCard` 增加优先级和异常态。
- `UModal` 支持长表单固定 footer。
- 增加统一 tooltip 组件用于禁用原因和图标解释。

### Phase 4：视觉打磨与验证

- 使用桌面宽度 1080x860、1366x900、1440x900 验证。
- 检查所有文字不溢出按钮和标签。
- 检查横向滚动表格固定列阴影 / 背景。
- 检查浅色主题下对比度。
- 检查主题色切换后所有 focus / active / tag 状态正常。

## 11. 验收标准

- 侧边栏和页面标题无不必要中英混用。
- 概览页首屏能明确看出服务是否运行、监听地址、供应商健康。
- 流量页无需横向滚动即可判断一条请求是否成功、耗时、路由、错误摘要。
- 供应商页无需横向滚动即可判断供应商是否可用。
- 所有空状态都有下一步操作。
- 危险操作都有确认弹窗。
- 所有图标按钮有可访问名称。
- 页面在 1080px 宽度下无明显内容重叠。
- 主题色和按钮尺寸设置仍然生效。

## 12. 推荐的第一轮改动清单

第一轮建议只做低风险优化：

1. 导航和页面文案统一中文。
2. 概览页模块顺序调整，突出服务状态与供应商健康。
