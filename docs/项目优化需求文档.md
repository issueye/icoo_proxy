# icoo_proxy 项目优化需求文档

> 文档版本：v1.0  
> 创建日期：2026-04-17  
> 适用范围：`icoo_proxy` 当前桌面管理端 + 本地 AI Gateway  
> 文档目标：基于现有实现，梳理新一阶段产品优化需求，为设计、开发、联调和验收提供统一依据。

---

## 1. 背景与目标

当前项目已经具备以下基础能力：

- 本地启动 AI Gateway，并对外暴露 OpenAI 兼容接口；
- 管理多个上游供应商；
- 配置模型映射、端点模式和基础路由；
- 支持协议转换、请求转发和请求日志审计；
- 在桌面端展示网关总览、供应商、日志、设置等页面。

但从当前实现看，**访问鉴权、供应商身份、上游端点、协议转换入口** 仍然是分散设计：

- 网关对外访问鉴权当前仍挂在 `GatewayConfig.AuthKey`，属于总览页/网关配置的一部分；
- API 密钥尚未成为独立管理实体，无法承载更细粒度的权限和关联关系；
- 端点模式当前只是供应商上的一个枚举字段，无法形成真正的“端点管理能力”；
- 协议转换主要依赖“供应商类型 + 端点模式”隐式推断，缺少可管理、可扩展、可配置的转换入口。

### 本轮优化目标

围绕“认证、上游配置、协议转换”三条主线，将系统从“供应商中心”升级为“API 密钥 + 端点 + 供应商”三元组合驱动的网关模型。

本轮目标包含四项：

1. 新增 **API 密钥管理模块**，并将网关总览页的访问鉴权迁移至该模块；
2. 支持 **API 密钥关联供应商**；
3. 新增 **端点管理模块**，用于管理请求使用的上游端点；
4. 通过 **端点 + API 密钥** 驱动不同 AI 协议之间的转换与路由。

## 2. 当前项目能力与问题分析

## 2.1 当前能力概览

结合现有实现，当前系统核心结构如下：

### 前端
- 首页：网关总览（运行状态、监听端口、访问鉴权 Key、启动停止等）
- 供应商页：供应商增删改查、连接测试、模型映射、默认模型
- 日志页：请求日志查看
- 设置页：主题与基础设置

### 后端
- `GatewayConfig` 当前包含：监听地址、默认供应商、日志级别、重试、`AuthKey`
- `ProviderConfig` 当前包含：类型、`APIBase`、`APIKey`、`EndpointMode`、优先级、模型映射等
- 请求处理逻辑大致为：
  - 接收 OpenAI 风格请求
  - 解析模型
  - 选择供应商
  - 根据供应商类型和端点模式决定适配器
  - 执行协议转换或透传
  - 记录审计日志

## 2.2 当前设计问题

### 问题 1：访问鉴权模型过于简单

当前网关访问鉴权只有一个全局 `AuthKey`：

- 无法区分不同调用方；
- 无法针对不同调用方绑定不同供应商或能力边界；
- 无法进行密钥停用、轮换、审计、分组管理；
- 无法承载未来的权限、配额、租户化扩展。

### 问题 2：供应商承担了过多职责

当前供应商对象同时承担：

- 上游身份；
- 上游地址；
- 上游 API Key；
- 上游端点模式；
- 路由目标；
- 协议适配入口。

这会导致两个问题：

1. 概念耦合：供应商、凭证、端点混在一起；
2. 扩展困难：一旦同一供应商下有多套密钥、多条端点、多协议能力，配置会迅速失控。

### 问题 3：端点只是枚举，不是资源

当前 `EndpointMode` 仅是一个字符串枚举，例如：

- `chat_completions`
- `responses`
- `anthropic_messages`
- `gemini_generate_content`

它只能表达“该供应商默认打哪个接口”，但无法表达：

- 一个供应商下有多个可选端点；
- 端点的请求协议与响应协议；
- 端点的 URL path、方法、能力标签、启用状态；
- 某 API 密钥只能访问某些端点；
- 某端点需要特定鉴权格式或转换链路。

### 问题 4：协议转换缺少显式配置边界

当前协议转换是后端内部逻辑，用户只能间接通过供应商类型感知转换行为，导致：

- 转换链路不透明；
- 端点粒度的协议控制能力不足；
- 无法表达“同一供应商不同端点对应不同协议”；
- 日志里难以解释真实发生了哪种协议转换。

## 3. 本轮需求范围

本轮需求只聚焦以下四项能力建设：

### In Scope
- API 密钥管理模块
- 网关访问鉴权迁移
- API 密钥与供应商关联
- 端点管理模块
- 端点与 API 密钥驱动的协议转换
- 配套的数据模型、后端接口、前端页面、日志字段调整
- 历史配置的兼容与迁移

### Out of Scope
- 多租户体系
- 配额/限流/计费
- RBAC 权限系统
- 第三方用户登录
- 复杂流量治理（熔断、权重、金丝雀）

这些能力可以为后续版本预留扩展点，但不在本轮交付范围内。

---

## 4. 核心设计原则

1. **资源解耦**：把 API 密钥、供应商、端点拆成独立资源；
2. **关系显式化**：通过关联关系代替隐式推断；
3. **向后兼容**：尽量兼容现有供应商和网关配置；
4. **最小迁移成本**：已有用户升级后应能自动迁移基础配置；
5. **转换可解释**：请求最终使用了哪个密钥、哪个端点、什么协议转换，应能在日志中体现；
6. **先满足 P0 业务闭环**：先跑通管理、路由、转换和验收，再考虑高级治理能力。

---

## 5. 目标能力模型

优化后，系统的核心资源建议调整为四层：

### 5.1 网关配置（Gateway Config）

只保留网关自身运行参数：

- 监听 Host/Port
- 默认行为参数
- 日志级别
- 重试策略

**不再承担 API 访问鉴权密钥本身的存储职责。**

### 5.2 API 密钥（ApiKey）

代表“客户端访问本网关时使用的身份凭证”。

职责：

- 控制谁可以访问网关；
- 限定该密钥可用的供应商；
- 限定该密钥可用的端点；
- 作为日志、审计和后续配额的主体。

### 5.3 供应商（Provider）

代表上游 AI 服务提供方或上游连接实例。

职责：

- 存储上游基础连接信息；
- 存储上游认证信息；
- 提供模型同步、连通性测试、健康状态；
- 作为端点所属主体。

### 5.4 端点（Endpoint）

代表一个可被路由到的上游请求入口。

职责：

- 描述具体请求地址与协议；
- 描述输入协议、输出协议、能力类型；
- 允许被 API 密钥引用；
- 参与请求转发和协议转换。

## 6. 详细需求整理

## 6.1 模块一：API 密钥管理模块

### 6.1.1 需求目标

将网关总览页面上的“访问鉴权”从单一配置项升级为完整的 API 密钥管理能力。

### 6.1.2 功能需求

| 编号 | 需求 | 优先级 | 说明 |
|---|---|---:|---|
| AK-001 | API 密钥列表 | P0 | 展示所有网关访问密钥及状态 |
| AK-002 | 新增 API 密钥 | P0 | 支持名称、密钥值/自动生成、描述、启用状态 |
| AK-003 | 编辑 API 密钥 | P0 | 支持修改名称、描述、状态、关联关系 |
| AK-004 | 删除 API 密钥 | P0 | 支持删除未使用的密钥 |
| AK-005 | 启用/禁用 API 密钥 | P0 | 禁用后请求立即失效 |
| AK-006 | 自动生成密钥 | P0 | 系统可自动生成安全随机值 |
| AK-007 | 脱敏展示 | P0 | 列表中仅显示前后缀 |
| AK-008 | 复制明文 | P1 | 用户可主动复制完整密钥 |
| AK-009 | 使用说明 | P0 | 展示如何在 `Authorization: Bearer <key>` 中使用 |
| AK-010 | 审计字段 | P1 | 记录创建时间、更新时间、最近使用时间 |

### 6.1.3 页面调整

#### 原网关总览页
- 删除或弱化“全局访问鉴权 Key”编辑能力；
- 保留“网关是否开启鉴权”的状态展示，但其数据来源改为 API 密钥模块；
- 提示用户前往 API 密钥管理模块维护访问凭证。

#### 新 API 密钥管理页
建议展示：

- 密钥名称
- 密钥前缀/后缀（脱敏）
- 状态（启用/禁用）
- 关联供应商数量
- 关联端点数量
- 最近使用时间
- 创建时间
- 操作：编辑 / 删除 / 复制 / 查看关联

### 6.1.4 业务规则

1. 网关收到请求后，优先校验请求头中的 Bearer Token；
2. 若系统存在已启用 API 密钥，则未带密钥或密钥错误的请求应被拒绝；
3. 若系统没有任何启用中的 API 密钥，可视为“未启用访问鉴权”；
4. 禁用或删除的 API 密钥不可再访问网关；
5. 每个请求在进入路由前，需要先解析出调用方 API 密钥身份。

## 6.2 模块二：API 密钥关联供应商

### 6.2.1 需求目标

让 API 密钥不只是“是否能访问网关”，还能够决定“能访问哪些供应商”。

### 6.2.2 功能需求

| 编号 | 需求 | 优先级 | 说明 |
|---|---|---:|---|
| AKP-001 | 关联多个供应商 | P0 | 一个 API 密钥可关联多个供应商 |
| AKP-002 | 供应商白名单控制 | P0 | 若设置了关联列表，则仅可访问白名单供应商 |
| AKP-003 | 无关联的默认行为 | P0 | 需明确空关联时是“全部可用”还是“不可用”，建议采用显式开关 |
| AKP-004 | UI 关联配置 | P0 | 在 API 密钥编辑页可勾选供应商 |
| AKP-005 | 请求侧约束 | P0 | 路由时若目标供应商不在允许范围内，则拒绝或继续尝试其他候选 |

### 6.2.3 建议规则

建议 API 密钥增加一个范围模式字段：

- `scopeMode = all`：可访问所有已启用供应商；
- `scopeMode = restricted`：仅可访问显式关联的供应商。

这样可以兼顾易用性和可控性。

### 6.2.4 路由影响

原来的路由流程是：

> 请求 → 解析模型 → 选供应商 → 转发

优化后应变为：

> 请求 → 校验 API 密钥 → 得到密钥可用范围 → 解析模型/规则 → 在允许范围内选供应商 → 转发

也就是说，**API 密钥会成为路由过滤条件的一部分**。

## 6.3 模块三：端点管理模块

### 6.3.1 需求目标

将原有 `EndpointMode` 从供应商字段升级为独立资源，使系统能明确管理“请求最终打到哪里、走什么协议、提供什么能力”。

### 6.3.2 端点定义建议

端点建议至少包含以下属性：

| 字段 | 说明 |
|---|---|
| `id` | 端点唯一标识 |
| `name` | 端点名称 |
| `providerId` | 所属供应商 |
| `path` | 上游路径，例如 `/v1/chat/completions` |
| `method` | 请求方法，通常为 `POST` / `GET` |
| `requestProtocol` | 入站目标协议，例如 `openai_chat`, `openai_responses`, `anthropic_messages`, `gemini_generate_content` |
| `responseProtocol` | 出站响应协议，通常与上游端点匹配 |
| `capability` | 能力类型，例如 `chat`, `responses`, `embeddings`, `models` |
| `enabled` | 是否启用 |
| `priority` | 候选优先级 |
| `isDefault` | 该供应商下的默认端点 |
| `remark` | 备注 |

### 6.3.3 功能需求

| 编号 | 需求 | 优先级 | 说明 |
|---|---|---:|---|
| EP-001 | 端点列表 | P0 | 查看所有端点 |
| EP-002 | 新增端点 | P0 | 指定所属供应商、协议、路径、能力 |
| EP-003 | 编辑端点 | P0 | 修改配置 |
| EP-004 | 删除端点 | P0 | 删除未使用端点 |
| EP-005 | 启用/禁用端点 | P0 | 禁用后不可参与路由 |
| EP-006 | 端点归属供应商 | P0 | 每个端点属于一个供应商 |
| EP-007 | 同供应商多端点 | P0 | 同一供应商可拥有多个端点 |
| EP-008 | 默认端点 | P1 | 每个供应商可设置某能力的默认端点 |
| EP-009 | 连接测试 | P1 | 验证端点可达性或最小请求能力 |
| EP-010 | 协议类型展示 | P0 | 在 UI 中直观看到端点协议和能力 |

### 6.3.4 页面建议

新增“端点管理”页面，列表支持展示：

- 端点名称
- 所属供应商
- 能力类型
- 请求协议
- 路径
- 状态
- 默认标识
- 优先级
- 操作：编辑 / 删除 / 测试 / 启停

供应商详情页也可嵌入“该供应商下的端点列表”作为子视图。

## 6.4 模块四：通过端点 + API 密钥实现协议转换

### 6.4.1 需求目标

让协议转换不再仅由供应商类型隐式决定，而是由：

- 客户端访问所使用的 API 密钥；
- API 密钥允许访问的端点；
- 端点定义的协议能力；
- 请求内容（模型、能力、规则）

共同决定最终转换和转发行为。

### 6.4.2 目标处理链路

建议新的请求处理流程如下：

```text
客户端请求
  → 校验 API 密钥
  → 根据 API 密钥得到允许的供应商/端点集合
  → 识别当前入站协议（网关入口通常为 OpenAI 兼容）
  → 根据模型/规则/能力选择候选端点
  → 在候选端点中选择最终端点
  → 根据 入站协议 -> 端点请求协议 执行请求转换
  → 调用上游
  → 根据 端点响应协议 -> 网关出站协议 执行响应转换
  → 返回结果
```

### 6.4.3 核心业务规则

| 编号 | 规则 | 优先级 | 说明 |
|---|---|---:|---|
| PCX-001 | API 密钥先于路由 | P0 | 未通过 API 密钥校验，不进入路由 |
| PCX-002 | API 密钥限制候选端点 | P0 | 只能从 API 密钥允许的端点中选目标 |
| PCX-003 | 端点声明协议 | P0 | 转换链路由端点协议显式决定 |
| PCX-004 | 同供应商可多协议 | P0 | 同一供应商可配置多个不同协议端点 |
| PCX-005 | 路由结果写入日志 | P0 | 记录请求命中的 API 密钥、端点、协议转换链 |
| PCX-006 | 无可用端点返回明确错误 | P0 | 返回“无可访问端点/无匹配端点”而非泛化失败 |

### 6.4.4 典型场景

#### 场景 A：OpenAI 请求转 Anthropic 端点

- 客户端调用 `/v1/chat/completions`
- 携带 API 密钥 `sk_xxx`
- `sk_xxx` 允许访问供应商 `anthropic-prod`
- 该供应商下存在端点：`anthropic_messages`
- 系统执行 `OpenAI Chat -> Anthropic Messages` 请求转换
- 上游返回后执行 `Anthropic -> OpenAI` 响应转换

#### 场景 B：同一供应商存在多个端点

例如某 OpenAI 兼容供应商同时支持：

- `/v1/chat/completions`
- `/v1/responses`

则端点管理模块可以把它们拆成两个端点，由不同 API 密钥或不同能力规则控制访问范围。

#### 场景 C：按密钥隔离访问能力

- `dev-key` 仅允许访问测试供应商和测试端点；
- `prod-key` 仅允许访问生产供应商和生产端点；
- 两者都通过同一个本地网关入口接入，但路由边界不同。

## 7. 数据模型调整建议

以下为建议的数据结构方向，最终字段名可按项目代码风格微调。

## 7.1 GatewayConfig 调整

### 当前

- `ListenHost`
- `ListenPort`
- `DefaultProvider`
- `LogLevel`
- `RetryCount`
- `RetryIntervalMs`
- `AuthKey`

### 建议调整后

- 保留运行参数；
- 移除 `AuthKey` 作为主配置字段；
- 若需要兼容迁移，可短期保留为废弃字段，仅用于首次迁移。

## 7.2 新增 ApiKeyConfig

```go
type ApiKeyConfig struct {
    ID                  string    `json:"id"`
    Name                string    `json:"name"`
    Key                 string    `json:"key"`
    Description         string    `json:"description"`
    Enabled             bool      `json:"enabled"`
    ScopeMode           string    `json:"scopeMode"` // all / restricted
    ProviderIDs         []string  `json:"providerIds"`
    EndpointIDs         []string  `json:"endpointIds"`
    LastUsedAt          time.Time `json:"lastUsedAt"`
    CreatedAt           time.Time `json:"createdAt"`
    UpdatedAt           time.Time `json:"updatedAt"`
}
```

说明：

- `ProviderIDs` 用于供应商白名单；
- `EndpointIDs` 用于端点白名单；
- 后续若只做供应商限制，也建议先预留 `EndpointIDs`，避免二次迁移。

## 7.3 ProviderConfig 调整

当前 `ProviderConfig` 中建议保留：

- `ID`
- `Name`
- `Type`
- `APIBase`
- `APIKey`
- `Enabled`
- `Priority`
- `ExtraConfig`
- `LLMs`
- `DefaultModel`

建议调整：

- `EndpointMode` 从“供应商核心字段”降级或废弃；
- 未来由独立 `EndpointConfig` 替代。

## 7.4 新增 EndpointConfig

```go
type EndpointConfig struct {
    ID               string `json:"id"`
    Name             string `json:"name"`
    ProviderID       string `json:"providerId"`
    Path             string `json:"path"`
    Method           string `json:"method"`
    Capability       string `json:"capability"`       // chat / responses / embeddings / models
    RequestProtocol  string `json:"requestProtocol"`  // openai_chat / openai_responses / anthropic_messages / gemini_generate_content
    ResponseProtocol string `json:"responseProtocol"`
    Enabled          bool   `json:"enabled"`
    Priority         int    `json:"priority"`
    IsDefault        bool   `json:"isDefault"`
    Remark           string `json:"remark"`
}
```

## 8. 后端改造需求

## 8.1 配置层

### 需求
- 配置服务新增 API 密钥和端点配置读写；
- 支持旧配置迁移；

- `ConfigProvider` 接口新增：
  - `GetApiKeys()`

## 8.2 网关鉴权层

### 需求
- 请求进入网关后先做 API 密钥校验；
- 支持 Bearer Token 解析；

### 关键改造点

## 8.3 路由与协议转换层

### 新增建议字段

- `apiKeyName`
- `endpointId`
- `endpointName`
- `requestProtocol`

## 10. 配置迁移需求

## 10.1 迁移目标

保证现有用户升级后可平滑进入新版本，不因配置结构调整导致网关不可用。

## 10.2 迁移规则建议

### 规则 1：旧 `GatewayConfig.AuthKey` 迁移为默认 API 密钥

如果旧配置存在 `AuthKey` 且非空：

- 自动生成一个 API 密钥记录，例如：
  - `name = Default Imported Key`
  - `key = old.AuthKey`
  - `enabled = true`
  - `scopeMode = all`
- 同时将 `GatewayConfig.AuthKey` 标记为废弃或清空。

### 规则 2：旧 `Provider.EndpointMode` 迁移为默认端点

对每个供应商：

- 根据其 `EndpointMode` 生成一条默认端点记录；
- 若 `EndpointMode` 为空，则按供应商类型推导默认端点；
- 生成后的端点标记为 `isDefault = true`。

### 规则 3：历史供应商仍可继续工作

即使用户不手动进入端点页配置，迁移后系统也应自动拥有最小可用端点集，保证旧供应商仍能正常路由。

## 11. 验收标准

## 11.1 API 密钥管理验收

- 可以新增、编辑、删除、启用、禁用 API 密钥；
- API 密钥支持脱敏展示；
- 携带正确 Bearer Token 可访问网关；
- 未携带或密钥错误时返回鉴权失败；
- 网关总览页不再直接维护全局 `AuthKey`。

## 11.2 API 密钥关联供应商验收

- API 密钥可配置供应商白名单；
- 请求命中白名单外供应商时不可转发；
- 在存在多个候选供应商时，仅在允许范围内选路由。

## 11.3 端点管理验收

- 可对端点做增删改查；
- 端点可归属供应商；
- 同一供应商可存在多个端点；
- 端点启用/禁用能实际影响路由；
- 旧 `EndpointMode` 可被迁移为默认端点。

## 11.4 协议转换验收

- 可通过端点协议配置驱动转换链路；
- OpenAI → Anthropic、OpenAI → Gemini 等已有能力不被破坏；
- 日志中可看到 API 密钥、端点、协议转换信息；
- 当无可用端点时返回明确错误。

## 13. 风险与注意事项

### 风险 1：配置迁移导致兼容问题
需要保证旧配置升级后至少能生成一套默认 API 密钥和默认端点，避免历史用户网关失效。

### 风险 2：路由逻辑复杂度上升
从“选供应商”升级为“选端点”，会让路由、日志、测试、异常提示都同步复杂化，需要统一抽象。

### 风险 3：前后端字段同步成本增加
Wails 绑定接口、Pinia store、后端配置结构会同时变更，建议先冻结字段定义再并行开发。

### 风险 4：协议适配器边界需重新整理
当前适配器较多依赖供应商类型，改造后要避免“端点协议”和“供应商类型”两套规则互相冲突。

---

## 14. 结论

这次优化的本质，不是简单新增两个页面，而是把当前项目的核心模型从：

> 网关配置 + 供应商

升级为：

> 网关配置 + API 密钥 + 供应商 + 端点

这样做的直接收益是：

- 访问控制从单一全局密钥升级为可管理的 API 密钥体系；
- 供应商不再承担凭证、端点、转换三类混合职责；
- 协议转换从隐式逻辑升级为显式端点能力；
- 为未来的权限、配额、租户、环境隔离留下稳定扩展点。

如果按当前项目阶段判断，这是一轮合理且必要的结构升级，建议按 **“先后端模型与迁移，再前端页面与路由，再联调验收”** 的顺序推进。
