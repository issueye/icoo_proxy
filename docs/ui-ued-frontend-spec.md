# icoo_proxy UI/UED 前端落地规范

> 文档版本：v1.0  
> 适用范围：`frontend/src` 内所有 Vue 页面、布局组件、基础 UI 组件与业务组件  
> 面向对象：前端开发、组件维护、页面重构、主题维护  
> 规范目标：统一整体 UI 风格、组件状态、布局密度、主题变量和交互细节，确保后续新增页面与现有桌面工具风格一致。

---

## 1. 总体设计定位

`icoo_proxy` 是一个桌面端 AI 网关管理工具，不是营销型 Web 页面，也不是通用后台模板。整体 UI 应保持：

- **桌面工具感**：结构稳定、控件克制、信息密度适中；
- **专业配置感**：突出状态、配置项、表格、日志、规则和表单；
- **低干扰视觉**：少渐变、少大色块、少装饰性动画；
- **高可读性**：正文、标签、状态、错误和数据都应清晰可扫读；
- **主题一致性**：所有颜色、圆角、阴影、间距必须优先来自全局 token。

视觉关键词：

- 清晰
- 克制
- 稳定
- 系统化
- 工具型
- 高密度但不拥挤

禁止方向：

- 不做强营销风格的大面积渐变；
- 不做玻璃拟态、霓虹发光、强投影；
- 不在单页里临时定义一套独立视觉体系；
- 不用过大的圆角和过重阴影表达层级；
- 不用彩色背景大面积包裹普通内容。

---

## 2. 主题系统规范

### 2.1 主题变量分层

项目当前主题变量集中在 `frontend/src/style.css`。后续开发必须遵守三层变量逻辑：

1. **基础语义 token**：`--ui-*`
2. **兼容语义 token**：`--color-*`
3. **组件样式类**：`.btn-*`、`.input-*`、`.toolbar-*`、`.settings-*` 等

新增样式应优先使用：

- 背景：`--ui-bg-*`
- 边框：`--ui-border-*`
- 文字：`--color-text-*` 或 `--ui-text-*`
- 强调色：`--color-accent` / `--ui-accent`
- 状态色：`--color-success` / `--color-warning` / `--color-error`
- 圆角：`--radius-*`
- 阴影：`--shadow-*`
- 间距：`--space-*`

禁止直接硬编码常用颜色，例如：

```css
/* 不推荐 */
color: #333;
border-color: #ddd;
background: #fff;
```

允许硬编码的例外：

- 特定品牌色且已被产品确认；
- 图表调色板；
- 第三方高亮主题覆盖；
- 极少数需要固定白色文字的主按钮，例如 `color: #fff`。

---

### 2.2 亮色主题基调

亮色主题是默认主题，应体现“Windows 桌面工具”的轻量感。

核心控制：

- 页面背景使用 `--ui-bg-app`；
- 窗口背景使用 `--ui-bg-window`；
- 面板使用 `--ui-bg-surface`；
- 次级面板、空态、工具栏使用 `--ui-bg-surface-muted` 或 `--ui-bg-toolbar`；
- 描边优先使用 `--ui-border-default`；
- 弱分割线使用 `--ui-border-subtle`。

亮色主题不要出现：

- 大面积纯蓝背景；
- 大面积高饱和状态色；
- 普通卡片使用深色边框；
- 内容面板使用过重阴影。

---

### 2.3 深色主题基调

深色主题不是反色，而是低亮度桌面工具主题。

要求：

- 保持层级差异，不能全部使用同一深灰；
- 边框可见但不刺眼；
- 强调蓝在深色下可以略亮；
- 状态色使用 soft 背景时必须控制透明度；
- 阴影只表达浮层，不用于普通卡片层级。

深色主题新增组件时必须检查：

- 文本是否低对比；
- disabled 是否仍可区分；
- hover 是否明显但不过亮；
- focus ring 是否清晰；
- 表格、抽屉、弹窗边界是否可见。

---

### 2.4 强调色使用规则

强调色 `--color-accent` 只用于：

- 主按钮；
- 当前导航项；
- 当前 Tab / Segment；
- 输入框 focus；
- 关键状态徽标；
- 链接或可点击强调文本；
- 进度条、选中态、复制成功等局部反馈。

不得用于：

- 大面积页面背景；
- 普通说明文字；
- 每个卡片标题；
- 普通图标默认色；
- 无交互意义的装饰边框。

## 3. 布局规范

### 3.1 应用骨架

应用整体保持固定桌面结构：

- 顶部标题栏：应用品牌、当前模块、窗口工具；
- 左侧侧边栏：一级导航；
- 主内容区：当前页面内容；
- 底部状态栏：网关运行状态、端口、桌面版本信息。

新增一级功能必须进入侧边栏，不允许在页面内部再做“伪一级导航”。

---

### 3.2 页面容器

页面根节点统一使用：

- `.app-page`：用于普通内容页；
- `.page-shell` + `.page-frame`：用于需要左右分栏或设置中心式结构的页面。

规则：

- 页面默认 padding 使用 `--page-padding`，当前为 `16px`；
- 页面内区块间距使用 `--section-gap`，当前为 `16px`；
- 页面不应自行设置大于 `24px` 的外层 padding；
- 不应在 `.app-page` 内再包一层全屏背景容器。

推荐结构：

```vue
<template>
  <div class="app-page feature-page">
    <PageHeader title="页面标题" />
    <section class="toolbar-surface">...</section>
    <div class="feature-workspace">...</div>
  </div>
</template>
```

---

### 3.3 页面头部

页面头部统一使用 `PageHeader`。

规范：

- 标题应短，通常 2-6 个中文字符；
- 操作按钮放入 `#actions`；
- 主操作在右侧第一个或最后一个位置保持一致，推荐最后一个为主按钮；
- 页面头部不承载长说明，长说明应放到页面内容区或空态中。

标题示例：

- `网关总览`
- `供应商管理`
- `对话规则`
- `请求日志`
- `设置中心`

---

### 3.4 工具栏

筛选、刷新、搜索、统计摘要统一放入 `.toolbar-surface`。

规则：

- 工具栏背景使用 `--ui-bg-toolbar`；
- padding 使用 `10px 12px`；
- 内部控件间距使用 `8px`；
- 左侧为输入/筛选；右侧为统计 chip；
- 不在工具栏中放复杂表单；
- 不在工具栏中放危险操作。

工具栏 chip 使用：

- `.toolbar-chip`
- `.info-chip`
- `.summary-chip`

chip 只展示短信息，例如：

- `共 12 条`
- `健康 3 条`
- `端口 16790`

---

### 3.5 分栏页面

常见结构：

- 左列表 + 右详情；
- 左设置导航 + 右设置内容；
- 主表格 + 右详情面板。

规范：

- 两栏间距统一 `16px`；
- 左栏宽度保持稳定，推荐 `264px`、`320px`、`340px`；
- 右侧详情区应可滚动；
- 左栏负责导航/列表，右栏负责编辑/详情，不要职责混杂；
- 小屏时允许单列堆叠。

---

## 4. 间距与密度规范

### 4.1 间距刻度

只能使用以下主间距：

- `2px`：极细微位置修正；
- `4px`：紧密元素间距；
- `6px`：标签与输入框；
- `8px`：行内控件；
- `10px`：紧凑内边距；
- `12px`：组内间距；
- `16px`：页面区块、面板内边距；
- `20px`：弹窗内容大间隔；
- `24px`：空态或大区块；
- `32px`：少量大空态。

不要新增 `7px`、`13px`、`18px` 这类非规范值，除非为兼容图标视觉居中。

---

### 4.2 页面密度

桌面端默认采用紧凑密度：

- 按钮高度：`32px`；
- 小按钮高度：`28px`；
- 大按钮高度：`36px`；
- 输入框高度：`32px`；
- Badge 高度：`22px`；
- 图标按钮：`32px`；
- 表格行高建议：`40px` - `48px`。

不要使用移动端风格的大控件密度，例如 44px 以上的常规按钮。

---

## 5. 字体与文案规范

### 5.1 字体

全局字体栈使用 `--font-ui`：

```css
--font-ui: "Segoe UI Variable", "Segoe UI", "Microsoft YaHei UI", "Microsoft YaHei", "PingFang SC", sans-serif;
```

等宽内容使用 `--font-mono`：

- API Base；
- 端口；
- 请求路径；
- JSON 片段；
- 日志 payload；
- curl 示例。

---

### 5.2 字号层级

默认字号层级：

- Display：`24px`，仅用于极少数页面主标题或大状态；
- Title 1：`20px`，用于设置分区标题；
- Title 2：`16px`，用于面板标题；
- Body：`13px`，用于主体内容；
- Body Small：`12px`，用于表单标签、说明；
- Caption：`11px`，用于 chip、kicker、辅助元信息。

页面中不得随意使用 `14px`、`15px` 作为新层级。确需使用时，应先判断是否能复用 `13px` 或 `16px`。

---

### 5.3 文案语气

界面文案应简短、明确、操作导向。

推荐：

- `启动网关`
- `停止网关`
- `刷新日志`
- `测试连接`
- `保存规则`
- `添加供应商`

避免：

- `点击这里来进行供应商连接可用性的测试`
- `系统正在为您努力加载中，请稍候`
- `哎呀，好像出了点问题`

错误文案必须说明可操作方向，例如：

- `API Base 不能为空`
- `连接失败，请检查 API Key 或网络代理`
- `未找到匹配模型，请确认模型映射配置`

---

## 6. 颜色规范

### 6.1 文本颜色

文本颜色按信息层级使用：

- 主标题、关键数据：`--color-text-primary`
- 普通正文、次级值：`--color-text-secondary`
- 辅助说明、placeholder、元信息：`--color-text-muted`
- 禁用文本：`--ui-text-disabled`

禁止用透明黑或透明白手写文本颜色。

---

### 6.2 状态色

状态色只用于有明确状态语义的位置。

成功：

- 运行中；
- 连接正常；
- 保存成功；
- 请求 2xx。

警告：

- 启用但健康异常；
- 配置不完整；
- 请求 4xx；
- 需要用户检查但不阻断。

错误：

- 删除危险操作；
- 网关启动失败；
- 请求 5xx；
- 表单校验失败；
- 连接失败。

信息：

- 当前选中；
- 普通提示；
- 鉴权已启用；
- 非风险性状态。

---

## 7. 组件规范

### 7.1 Button 按钮

按钮统一使用 `.btn` 体系或 `Button.vue`。

#### 类型

- `.btn-primary`：页面主操作；
- `.btn-secondary`：次级操作；
- `.btn-danger`：危险确认操作；
- `.btn-success`：启动类正向动作；
- `.btn-ghost`：低权重操作；
- `.btn-link`：文本链接操作；
- `.btn-icon` / `.icon-btn`：仅图标操作。

#### 使用规则

- 单个操作区最多一个 primary；
- 删除、关闭、清空等不可逆动作必须使用 danger；
- 普通刷新、复制、取消使用 secondary；
- 表格行内编辑、删除、测试优先使用 `icon-btn`；
- 图标按钮必须提供 `title` 或可访问名称；
- 禁用态必须真实设置 `disabled`，不要只改颜色。

#### 尺寸

- 默认：`32px`
- 小型：`28px`
- 大型：`36px`
- 图标：`32px * 32px`

#### 状态

每个按钮必须覆盖：

- default；
- hover；
- active；
- focus-visible；
- disabled；
- loading。

加载态建议：

- 文案从 `保存` 变为 `保存中...`；
- 禁用按钮避免重复提交；
- 可配合旋转图标，但不强制。

---

### 7.2 Input 输入框

输入框统一使用 `Input.vue` 或全局输入类。

#### 基础规则

- 高度默认 `32px`；
- 圆角 `--radius-sm`；
- 边框 `--ui-border-default`；
- focus 使用 `--shadow-focus`；
- label 位于输入框上方；
- description 和 error 位于输入框下方。

#### 状态

必须覆盖：

- default；
- hover；
- focus；
- error；
- disabled；
- readonly；
- placeholder。

#### 错误态

错误态规则：

- 边框使用 `--ui-danger`；
- 辅助说明使用 `--color-error`；
- 错误文案必须紧贴对应字段；
- 不只用 toast 表达字段错误。

#### 密码输入

密码输入应支持：

- 显示/隐藏切换；
- 编辑供应商时 API Key 留空代表保留原值；
- placeholder 说明规则，而不是显示真实 key。

---

### 7.3 Select 下拉选择

Select 统一使用 `Select.vue`。

使用规则：

- 单选用于类型、状态、端点模式、默认模型；
- 多选只在确有批量选择需求时使用；
- 选项超过 8 个时开启搜索；
- placeholder 使用 `请选择...` 或明确对象，例如 `请选择供应商`；
- 下拉宽度不得小于触发器宽度；
- 禁用项必须不可点击且视觉弱化。

Select 状态必须覆盖：

- closed；
- open；
- selected；
- hover option；
- disabled option；
- empty search；
- error。

---

### 7.4 StatusBadge 状态徽标

状态显示统一使用 `StatusBadge.vue`。

支持状态：

- `neutral`
- `success`
- `warning`
- `error`
- `info`

使用规则：

- 状态文字必须短，例如 `正常`、`异常`、`禁用`、`运行中`；
- 列表中优先使用 badge，而不是整行染色；
- 状态点颜色必须与状态文字一致；
- loading 时使用旋转图标，不显示状态点；
- badge 不承载长句说明。

---

### 7.5 Card / Panel 面板

面板用于承载一组相关内容。

统一样式：

- 边框：`--ui-border-default`
- 圆角：`--radius-md`
- 背景：`--ui-bg-surface`
- 阴影：`--shadow-rest`
- padding：`16px`

使用规则：

- 普通页面不要过度卡片化；
- 避免卡片套卡片；
- 一个面板只表达一个主题；
- 面板标题用 `.section-title`；
- 面板说明用 `.panel-description`；
- 面板右上角可放 badge 或轻量按钮。

---

### 7.6 DataTable 表格

表格适合：

- 供应商列表；
- 请求日志；
- 模型映射；
- 规则列表中的结构化数据。

规范：

- 表头文字使用 `12px` 或 `13px`，加粗；
- 行内主值使用 `--color-text-primary`；
- 行内次值使用 `--color-text-muted`；
- 操作列固定右侧；
- 数字、端口、路径、模型名可使用等宽字体；
- 状态列使用 `StatusBadge`；
- 点击行打开详情时，hover 必须明显。

表格空态：

- 使用 `.empty-state`；
- 文案说明为什么为空；
- 如果有明确下一步，提供一个主按钮。

---

### 7.7 Dialog 弹窗

弹窗用于需要用户确认或短表单编辑的场景。

规则：

- 危险确认必须使用弹窗；
- 弹窗标题必须明确操作对象；
- footer 左侧可放辅助动作，右侧放取消和确认；
- 确认按钮根据动作语义选择 primary 或 danger；
- 弹窗内不放复杂长列表，复杂编辑使用 Drawer。

按钮顺序：

- `取消` 在左；
- `确认 / 保存 / 删除` 在右；
- 危险确认按钮使用 `btn-danger`。

---

### 7.8 Drawer 抽屉

抽屉用于复杂编辑，不打断主列表上下文。

适合：

- 添加/编辑供应商；
- 模型映射配置；
- 规则详细编辑；
- 大字段配置。

规则：

- 宽度建议 `560px` 或 `640px`；
- header 显示标题；
- summary 区展示当前对象状态；
- body 可滚动；
- footer 固定操作区；
- 保存时禁用确认按钮；
- 关闭前如有未保存改动，后续应补确认机制。

---

### 7.9 Toast 通知

Toast 用于操作结果反馈。

适合：

- 保存成功；
- 复制成功；
- 刷新完成；
- 测试连接成功/失败；
- 非字段级错误。

不适合：

- 表单字段校验；
- 需要用户长时间阅读的错误；
- 需要明确选择的确认操作。

Toast 文案应短：

- `已保存`
- `已复制 API Base`
- `连接失败，请检查配置`

---

## 8. 业务页面规范

### 8.1 网关总览页

核心任务：确认网关是否可用，并快速复制接入信息。

信息优先级：

1. 运行状态；
2. API Base；
3. 供应商健康度；
4. 可用模型数量；
5. 鉴权状态；
6. 调用示例。

规范：

- 启动/停止按钮必须醒目；
- API Base 使用等宽字体；
- 鉴权状态使用 badge；
- 诊断项使用短句，不写长说明；
- curl 示例放在代码块中，允许复制。

---

### 8.2 供应商管理页

核心任务：管理上游供应商和模型映射。

规范：

- 列表优先表格；
- 筛选放工具栏；
- 行内操作保持顺序：测试、模型、编辑、删除；
- 删除必须确认；
- API Key 输入用 password；
- 编辑时 API Key 留空保留旧值；
- 连接测试结果用 toast + 状态更新表达。

供应商状态：

- 健康且启用：`success / 正常`
- 启用但不健康：`warning / 异常`
- 未启用：`error 或 neutral / 禁用`

---

### 8.3 对话规则页

核心任务：配置路由规则。

规范：

- 左侧规则列表，右侧规则详情；
- 规则列表展示名称、启用状态、匹配方式、优先级、pattern；
- 右侧编辑区按基础信息、匹配条件、路由目标分组；
- 保存时过滤无效规则可以保留，但 UI 应提示哪些字段必填；
- `pattern` 和 `providerId` 是有效规则的最低要求。

匹配方式文案统一：

- `模型名匹配`
- `System 包含`
- `任意消息包含`
- `用户消息包含`
- `助手消息包含`

---

### 8.4 请求日志页

核心任务：排查请求链路。

规范：

- 左侧或主区展示日志表格；
- 右侧展示详情；
- 状态码使用 `StatusBadge`；
- 请求路径、模型、耗时应可快速扫读；
- payload 使用等宽字体和 JSON 格式化；
- 失败筛选放工具栏；
- 默认选中第一条日志。

日志详情优先级：

1. 状态码和错误信息；
2. 供应商与目标模型；
3. 上游地址；
4. 请求体；
5. 响应头；
6. 响应体。

---

### 8.5 设置页

设置页用于低频配置，不应抢占核心网关操作。

规范：

- 左侧为设置分类；
- 右侧为当前分类内容；
- 外观设置可以使用 segment、swatch、preview；
- 关于页面只展示版本、技术栈、简要说明；
- 不在设置页重复供应商和规则配置。

---

## 9. 交互状态规范

### 9.1 Loading

Loading 应根据区域大小选择表达：

- 按钮内：按钮文案变为 `保存中...` / `测试中...`；
- 表格：使用表格 loading 或骨架；
- 页面首次加载：可显示空态式加载；
- 局部刷新：保留旧数据，只让刷新按钮旋转。

禁止：

- 整页无差别遮罩；
- 所有刷新都清空旧数据；
- loading 文案过长。

---

### 9.2 Empty

空态必须回答三个问题：

1. 当前为空的对象是什么；
2. 为什么为空或什么时候会出现；
3. 用户下一步可以做什么。

示例：

- `暂未配置供应商`
- `先添加第一个供应商后，才能配置模型映射与统一路由策略。`
- `添加第一个供应商`

---

### 9.3 Error

错误分为三类：

- 字段级错误：显示在字段下方；
- 操作级错误：toast；
- 页面级错误：面板内 alert 或 empty-state。

错误文案必须具体，不只写 `失败`。

推荐：

- `端口被占用，请更换监听端口后重试`
- `供应商连接失败，请检查 API Base 和 API Key`
- `模型不能为空`

---

### 9.4 Disabled

禁用态用于：

- 操作不可用；
- loading 期间防止重复提交；
- 缺少必要前置条件；
- 权限或环境不满足。

禁用态必须同时满足：

- 设置 `disabled`；
- 视觉弱化；
- 不响应 hover；
- 如原因不明显，应提供 tooltip 或说明。

---

### 9.5 Focus 与键盘可达

所有可交互元素必须有 `focus-visible`。

当前项目统一使用 `--shadow-focus`，新增组件必须复用。

要求：

- 按钮、输入框、下拉、tab、segment、图标按钮均可键盘聚焦；
- 不用 `outline: none` 后不补 focus 样式；
- 抽屉和弹窗打开后，后续应逐步补充焦点管理；
- 图标按钮必须有 `title` 或 aria-label。

---

## 10. 动效规范

动效用于确认状态变化，不用于装饰。

推荐时长：

- hover / color：`0.14s`
- fade：`0.16s`
- 页面切换：`0.16s - 0.2s`
- 小位移：`4px - 6px`

禁止：

- 大幅弹跳；
- 长于 `300ms` 的常规动效；
- 多个方向同时运动；
- 列表大量元素逐个入场动画。

现有 `.fade`、`.fade-slide`、`.spinning` 等应作为主要动效来源。

---

## 11. 图标规范

项目使用 `lucide-vue-next`。

规则：

- 常规图标尺寸：`14px - 16px`；
- 页面标题图标：`16px - 18px`；
- 空态图标：`32px - 40px`；
- 图标线条不单独改粗；
- 图标颜色默认继承文本色；
- 不在同一按钮内使用两个图标；
- 不混用其他图标库，除非明确补齐缺失图标。

常用映射：

- 网关：`Server` / `Network`
- 供应商：`Cpu`
- 日志：`ScrollText`
- 规则：`MessageSquareText`
- 设置：`Settings`
- 刷新：`RefreshCw`
- 新增：`Plus`
- 编辑：`Pencil`
- 删除：`Trash2`
- 测试：`Zap`
- 模型：`Database`

---

## 12. 可访问性规范

最低要求：

- 文本与背景满足基本对比度；
- 不只依赖颜色表达状态，必须配合文字或图标；
- 表单字段必须有 label；
- 图标按钮必须有 `title` 或 `aria-label`；
- 错误信息靠近对应字段；
- 键盘能访问主要操作；
- focus 样式清晰可见。

对于日志、模型名、路径等长文本：

- 列表中允许截断；
- 详情中必须可完整查看；
- 关键值推荐支持复制。

---

## 13. CSS 编写规范

### 13.1 优先级

新增样式优先级：

1. 复用已有组件；
2. 复用全局类；
3. 在页面 scoped style 中写页面局部布局；
4. 只有跨页面复用时才新增全局类。

不要为了一个页面写一套新的按钮、输入框、badge、card。

---

### 13.2 命名

页面局部类推荐：

- `feature-view`
- `feature-toolbar`
- `feature-workspace`
- `feature-panel`
- `feature-detail`
- `feature-empty`

组件状态类推荐：

- `.is-active`
- `.is-disabled`
- `.is-error`
- `.is-loading`
- `.is-running`

不要混用：

- `.active`
- `.selected`
- `.current`

除非组件已有历史实现。

---

### 13.3 Scoped 样式边界

页面内 `scoped` 样式只处理：

- 当前页面布局；
- 当前页面特殊网格；
- 当前页面局部视觉修正。

不应在页面 scoped 中重新定义：

- `.btn`
- `.input`
- `.status-badge`
- `.toolbar-surface`
- `.empty-state`
- 全局 token。

---

## 14. 新增页面验收清单

新增或重构页面完成前，必须检查：

- 是否使用 `.app-page` 或 `.page-shell`；
- 是否使用 `PageHeader`；
- 筛选区是否使用 `.toolbar-surface`；
- 主操作是否只有一个 primary；
- 危险操作是否使用 danger 并二次确认；
- 输入框、Select、按钮是否覆盖 disabled/loading/error；
- 空态是否有说明和下一步；
- 请求失败是否有可理解错误文案；
- 深色主题是否可读；
- focus-visible 是否存在；
- 图标按钮是否有 title；
- 是否避免了硬编码颜色；
- 是否复用了全局 token。

---

## 15. 主题风格把控红线

以下情况视为违反 UI/UED 规范：

- 页面级大面积使用渐变背景；
- 自定义一套与全局不同的按钮样式；
- 使用非 token 颜色构建主题；
- 在普通卡片上使用重阴影；
- 一屏内出现多个 primary 主按钮竞争；
- 只用颜色表达成功/失败；
- 表单错误只用 toast，不在字段处提示；
- 图标按钮无 title / aria-label；
- 深色主题下文字或边界不可读；
- 新增页面没有 loading、empty、error 设计。

---

## 16. 推荐落地顺序

后续 UI 优化或新增功能建议按以下顺序执行：

1. 先复用现有 token，不新增颜色；
2. 先搭建页面骨架，再填业务内容；
3. 先完成 default 状态，再补 hover / focus / disabled；
4. 先完成空态和错误态，再接真实数据；
5. 最后检查深色主题和键盘可达。

该顺序能减少视觉漂移，也能避免后期反复修样式。

---

## 17. 总结

`icoo_proxy` 的 UI 应持续保持“专业桌面配置工具”的一致风格。前端开发时，所有页面和组件都应围绕以下标准落地：

- 统一 token；
- 统一控件；
- 统一间距；
- 统一状态；
- 统一主题；
- 统一交互反馈。

这份规范后续应作为新增页面、组件改造、主题调整和 UI 评审的基准文档。
