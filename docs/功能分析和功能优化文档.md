# icoo_proxy 功能分析与功能优化文档

## 1. 项目定位

`icoo_proxy` 是一个基于 `Wails + Vue 3 + Go` 的桌面端 AI Gateway 管理工具，核心目标是：

- 以本地统一入口对外暴露 OpenAI 兼容接口；
- 聚合多个上游 AI 供应商；
- 通过模型映射、路由规则、协议适配完成请求转发；
- 提供桌面化的配置、诊断、日志与运维界面。

从当前实现看，这个项目不是单纯的“代理转发器”，而是一个**本地 AI 接入网关控制台**。

---

## 2. 当前项目功能分析

## 2.1 整体架构

项目分为两层：

### 前端桌面管理层
- 技术栈：`Vue 3 + Vite + Pinia + Vue Router`
- 作用：提供网关管理界面、供应商配置、对话规则配置、日志查看、主题设置

### 后端网关服务层
- 技术栈：`Go + Wails`
- 作用：启动本地 HTTP 网关、维护配置、管理供应商、做协议转换、执行路由决策、记录请求审计日志

### 当前具备的关键技术能力
- 本地桌面应用运行
- 本地 HTTP 网关监听端口
- OpenAI Chat Completions / Responses 兼容入口
- 多供应商配置与健康状态管理
- 模型别名与目标模型映射
- 基于规则的路由分发
- 请求/响应日志落库与展示
- 本地配置持久化

## 2.2 已实现功能模块

## A. 网关总览模块

### 当前功能
- 展示网关运行状态、监听端口、API Base
- 支持启动/停止网关
- 展示供应商健康度与可用数量
- 支持访问鉴权 Key 配置
- 提供调用示例和一键复制
- 自动轮询刷新状态

### 价值
- 降低用户接入门槛
- 让用户快速知道“网关是否可用、地址是什么、鉴权是否开启”
- 适合作为产品首页和运行态总控页

### 当前问题
- 运行态信息偏基础，缺乏真正的“运维总览”能力
- 没有近一段时间的请求量、错误率、平均延迟等关键指标
- “可用”只展示静态状态，不展示失败原因和最近检测时间
- 诊断提示更多是规则判断，不够像真实可执行诊断

---

## B. 供应商管理模块

### 当前功能
- 新增 / 编辑 / 删除供应商
- 供应商类型选择
- API Base、API Key、优先级、端点模式配置
- 启用/禁用供应商
- 测试供应商连接
- 为供应商配置模型映射
- 指定默认模型
- 展示健康状态、模型数量、优先级

### 价值
- 这是项目最核心的资源配置模块
- 使网关具备多上游聚合能力
- 模型映射能力能屏蔽上游差异，对客户端暴露统一模型名

### 当前问题
- 供应商测试结果较单薄，缺少耗时、错误分类、建议修复动作
- 模型映射是“手工输入”模式，批量维护成本高
- 缺少“从上游拉取模型并自动生成映射”的高效流程
- 没有供应商分组、标签、环境区分（开发 / 测试 / 生产）
- 缺少回退链路和故障转移可视化

---

## C. 对话规则模块

### 当前功能
- 新增 / 删除 / 编辑路由规则
- 支持多种匹配方式：
  - 模型名匹配
  - system 包含
  - 任意消息包含
  - 用户消息包含
  - 助手消息包含
- 可配置目标供应商、目标模型、优先级、启用状态
- 支持保存全部规则

### 价值
- 这是网关“智能分发”的核心能力
- 可以按模型名、上下文内容将请求路由到不同供应商
- 为后续扩展成本控制、场景路由、质量策略打下基础

### 当前问题
- 规则能力偏基础，仍属于“字符串匹配规则”阶段
- 缺少规则命中预览、冲突检测、优先级解释
- 用户无法直观看出某条请求为什么命中某条规则
- 缺少测试沙盒，无法输入样例请求验证规则结果
- 缺少规则模板与常见场景预设

---

## D. 请求日志模块

### 当前功能
- 查看请求日志列表
- 按条数拉取最近日志
- 仅查看失败日志
- 查看日志详情：
  - 请求路径
  - 供应商
  - 原始模型 / 目标模型
  - 状态码
  - 延迟
  - 来源 IP
  - 请求模式
  - 上游目标
  - 请求体 / 响应头 / 响应体
  - 错误信息

### 价值
- 是排障、验收、回归验证的重要模块
- 对网关产品来说，日志是非常关键的“可观测性入口”

### 当前问题
- 日志筛选维度不足，缺少按时间、模型、供应商、状态码过滤
- 缺少搜索能力，排查效率有限
- 没有聚合分析视图，例如失败 Top、最慢请求 Top
- 没有日志导出、脱敏导出能力
- 日志保留策略和容量管理未体现

---

## E. 设置模块

### 当前功能
- 外观主题切换
- 关于信息展示

### 价值
- 提供基础桌面应用设置能力
- 改善视觉体验

### 当前问题
- 设置项明显偏少，产品层面仍未形成“系统设置中心”
- 缺少以下关键设置：
  - 网关行为设置
  - 日志保留设置
  - 启动行为设置
  - 数据导入导出
  - 备份恢复
  - 安全设置

---

## F. 后端协议与网关能力

### 当前功能
- 启动本地网关监听端口
- 支持 `/v1/chat/completions`
- 支持 `/v1/responses`
- 支持 OpenAI 兼容上游直通
- 支持非 OpenAI 协议适配与转换
- 支持流式与非流式请求
- 支持模型映射
- 支持路由决策
- 支持请求日志审计落库
- 支持本地数据库配置持久化
- 支持旧配置迁移

### 价值
- 已具备一个 AI Gateway 产品的基本闭环
- 具备进一步扩展为“统一 AI 接入层”的基础

### 当前问题
- 缺少熔断、重试策略可视化
- 缺少限流、配额、租户隔离等网关级能力
- 缺少结构化监控指标输出
- 缺少请求链路追踪 ID
- 缺少配置版本管理和回滚能力

---

## 3. 当前产品能力总结

从产品角度看，当前版本已经覆盖了一个 AI Gateway 的第一阶段闭环：

1. **可接入**：能统一暴露接口；
2. **可配置**：能管理多个供应商；
3. **可路由**：能按规则把请求分发出去；
4. **可观察**：能查看请求日志；
5. **可使用**：桌面界面完整，基础交互已具备。

但当前仍然更偏向“工程可用版本”，距离“产品成熟版本”还有几个明显缺口：

- 智能化不足：规则与路由能力还不够强；
- 可观测性不足：运维指标与分析能力偏弱；
- 运维能力不足：缺少回退、导出、备份、容量策略；
- 企业化能力不足：缺少权限、配额、租户、环境隔离；
- 易用性不足：部分配置需要用户理解底层概念。

## 4. 功能优化建议

以下建议按产品价值和落地优先级划分。

## 4.1 P0：优先级最高，建议尽快做

### 1）增加“规则命中调试器”

#### 建议功能
在“对话规则”页增加一个测试面板：
- 输入模型名
- 输入 system / user message 示例
- 点击“模拟路由”
- 返回：
  - 命中的规则
  - 命中的理由
  - 目标供应商
  - 最终目标模型
  - 若未命中则显示默认路由结果

#### 产品价值
- 大幅降低规则配置门槛
- 减少“规则为什么没生效”的排障成本
- 这是路由产品的核心体验能力

---

### 2）增强日志检索与筛选

#### 建议功能
增加以下筛选项：
- 时间范围
- 供应商
- 模型
- 状态码区间
- 请求路径
- 是否流式
- 关键字搜索

增加以下能力：
- 复制日志详情
- 导出日志（JSON / CSV）
- 响应内容脱敏导出

#### 产品价值
- 直接提升排障效率
- 日志模块从“查看列表”升级为“诊断工具”

---

### 3）供应商模型自动同步

#### 建议功能
在供应商管理中增加：
- 从上游拉取模型列表
- 自动生成映射初稿
- 自动识别新增 / 删除 / 变更模型
- 支持一键同步与差异确认

#### 产品价值
- 显著降低维护成本
- 避免人工输入模型名造成错误
- 让多供应商接入更可持续

---

### 4）首页增加运行指标卡片

#### 建议功能
在网关首页新增：
- 最近 5 分钟请求数
- 最近 5 分钟失败率
- 平均响应时延
- 最慢供应商
- 最近错误类型 Top 3

#### 产品价值
- 首页从“配置页”升级为“监控总览页”
- 能更快发现系统异常

---

## 4.2 P1：高价值优化，建议第二阶段做

### 5）增加供应商故障转移 / 回退策略

#### 建议功能
- 为每个供应商配置失败后的回退供应商
- 支持按错误类型决定是否回退

#### 产品价值
- 提升网关稳定性

---

### 6）规则冲突检测与优先级解释

- 当前请求命中哪条规则

#### 产品价值
- 这是规则系统产品化的重要一步

## 7. 产品经理结论

当前项目已经具备一个 AI Gateway 桌面管理工具的基础产品形态，核心能力完整，工程实现路径也比较清晰。

它现在最主要的问题不是“缺功能入口”，而是：
- 关键链路的可解释性不够；
- 运维与排障能力还不够强；
- 高级网关能力还未产品化。

因此优化方向不建议继续横向铺很多新页面，而应该优先围绕三件事展开：

1. **让路由结果更可解释**；
2. **让日志更可诊断**；
3. **让供应商管理更自动化、更稳定**。

只要这三块做扎实，`icoo_proxy` 的产品完成度会明显上一个台阶。
