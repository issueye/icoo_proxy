# icoo_proxy 开发计划文档

> 版本：v1.0  
> 日期：2026-04-17  
> 目标：基于《项目优化需求文档》，按可执行顺序推进 API 密钥、端点、协议转换改造。

---

## 1. 开发目标

本轮采用**不兼容式修改**，直接把系统核心模型从：

- GatewayConfig + Provider

切换为：

- GatewayConfig + ApiKey + Provider + Endpoint

本次开发优先目标：

1. 后端先完成新配置模型与持久化；
2. 网关鉴权从 `GatewayConfig.AuthKey` 切换到 `ApiKey`；
3. 路由结果从“供应商”逐步升级到“端点”；
4. 前端补齐 API 密钥管理页、端点管理页；
5. 最后联调协议转换、日志与迁移逻辑。

---

## 2. 实施策略

采用四阶段推进，避免一次性改动过大导致系统完全不可运行。

### 阶段 1：配置与鉴权底座

目标：先让新资源模型存在，并让网关用新 API 密钥鉴权。

范围：

- 新增 `ApiKeyConfig`
- 新增 `EndpointConfig`
- 配置服务支持 API 密钥、端点的读写持久化
- 旧 `AuthKey` 向 `ApiKey` 迁移
- 网关鉴权中间件改为读取 API 密钥
- 补充配置层测试

交付结果：

- 系统不再依赖旧全局 `AuthKey` 完成访问控制
- 本地数据库可保存 API 密钥与端点

### 阶段 2：路由与后端服务改造

目标：把后端路由主语从 Provider 切到 Endpoint。

范围：

- `provider.Manager` 支持端点加载与解析
- `ResolveRequest()` 返回 `Endpoint + Provider`
- 路由时叠加 API 密钥范围过滤
- 请求日志增加 API 密钥/端点字段
- 协议转换依据端点协议字段执行

交付结果：

- 路由链路具备“鉴权身份 → 候选端点 → 最终端点”的能力

### 阶段 3：前端管理界面

目标：让桌面端具备新资源的完整管理能力。

范围：

- 新增 API 密钥管理页
- 新增端点管理页
- 调整网关总览页，移除旧 AuthKey 配置入口
- 调整供应商页，弱化 `EndpointMode`
- 新增对应 Pinia store 和 Wails 绑定接口

交付结果：

- 用户可在 UI 中管理 API 密钥、端点、供应商关联关系

### 阶段 4：联调、迁移、验收

目标：收口功能闭环。

范围：

- 验证配置迁移
- 验证 OpenAI → Anthropic / Gemini 转换不退化
- 验证 API 密钥白名单、端点白名单生效
- 验证日志审计字段完整
- 修复联调问题

交付结果：

- 满足优化需求文档中的 P0 验收项

---

## 5. 风险控制

### 风险 1：直接不兼容改动导致旧逻辑失效
处理方式：
- 配置层保留一次性迁移逻辑；
- 第一阶段优先保证网关仍可启动。

### 风险 2：前后端接口暂时不同步
处理方式：
- 本回合优先改后端底座；
- 前端待下一阶段集中补齐。

### 风险 3：端点改造范围大
处理方式：
- 先把端点配置落库；
- 下一阶段再切换路由主语。
