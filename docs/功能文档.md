# icoo_proxy 功能文档

## 1. 项目定位

`icoo_proxy` 是一个基于 **Wails + Go + Vue 3** 构建的桌面应用，用于在本地提供一个 **OpenAI 兼容的统一 AI 网关**。

它的核心目标是：

- 把多个上游 AI 服务供应商统一接到一个本地入口；
- 对外暴露统一的 OpenAI 风格接口；
- 支持模型映射、规则路由、请求转发、健康检查和请求日志；
- 通过桌面 UI 完成配置、调试和运维操作。

默认监听地址为：`127.0.0.1:16790`

默认对外接口前缀为：`/v1`

---

## 2. 整体架构

项目由两部分组成：

### 2.1 桌面端壳层

- 使用 `Wails` 将 Go 后端和 Vue 前端打包为桌面应用；
- 前端通过 Wails 绑定调用 Go 方法；
- 应用为无边框桌面窗口，包含自定义标题栏、侧边导航和状态栏。

### 2.2 本地 AI Gateway

Go 后端启动一个仅监听本机的 HTTP 服务：

- 监听地址：`127.0.0.1:<port>`
- 当前实现的网关接口：
  - `POST /v1/chat/completions`
  - `POST /v1/responses`
  - `GET /v1/models`
  - `GET /v1/health`

该网关负责：

- 接收客户端请求；
- 解析请求中的模型与消息内容；
- 根据规则决定走哪个供应商；
- 必要时完成协议转换；
- 返回统一格式结果；
- 记录请求审计日志。

## 3. 主要功能清单

### 3.1 网关启动与停止

应用启动时会自动：

- 注册协议适配器；
- 初始化配置存储；
- 初始化请求日志存储；
- 从配置中加载供应商；
- 启动本地网关服务；
- 异步刷新供应商模型列表。

UI 中也支持手动：

- 启动网关；
- 停止网关；
- 刷新网关状态。

状态信息包括：

- 是否运行；
- 当前端口；
- 供应商总数；
- 健康供应商数量。

---

### 3.2 统一 OpenAI 兼容接口

#### 3.2.1 `POST /v1/chat/completions`

功能：

- 接收 OpenAI Chat Completions 风格请求；
- 从请求体中提取 `model`；
- 转换为内部统一请求结构；
- 根据路由规则选择上游；
- 对 OpenAI 兼容供应商走透传；
- 对 Anthropic、Gemini 等非 OpenAI 协议供应商走协议转换；
- 支持流式和非流式响应。

#### 3.2.2 `POST /v1/responses`

功能：

- 接收 OpenAI Responses API 风格请求；
- 解析 `input` / `model` 等字段；
- 走同样的供应商选择与模型映射逻辑；
- 支持流式和非流式结果；
- 输出尽量保持 OpenAI Responses 风格。

#### 3.2.3 `GET /v1/models`

功能：

- 汇总当前已加载供应商模型；
- 返回 OpenAI 兼容格式的模型列表；
- 用于客户端检查网关当前可见模型。

#### 3.2.4 `GET /v1/health`

功能：

- 返回网关运行状态；
- 返回当前端口；
- 返回供应商数量与健康数；
- 用于本地探活与诊断。

---

### 3.3 供应商管理

桌面端支持对上游 AI 供应商进行完整管理。

每个供应商支持的配置项包括：

- `ID`
- 名称 `Name`
- 类型 `Type`
- `API Base`
- `API Key`
- 端点模式 `EndpointMode`
- 是否启用 `Enabled`
- 优先级 `Priority`
- 模型映射列表 `LLMs`
- 默认模型 `DefaultModel`

当前代码中明确支持的供应商类型：

- `openai`
- `anthropic`
- `gemini`

支持的供应商相关能力：

- 添加供应商；
- 编辑供应商；
- 删除供应商；
- 测试供应商连接；
- 查看供应商健康状态；
- 查看供应商类型与端点模式；
- 为每个供应商设置模型映射与默认模型。

---

### 3.4 端点模式选择

不同供应商可以指定请求应该打到哪个上游接口。

当前代码内置的端点模式包括：

- `chat_completions`
- `responses`
- `anthropic_messages`
- `gemini_generate_content`

默认归一化规则：

- `anthropic` 默认走 `anthropic_messages`
- `gemini` 默认走 `gemini_generate_content`
- 其他默认走 `chat_completions`

这使得系统可以把统一的内部请求映射到不同厂商的原生接口。

---

### 3.5 模型映射

项目支持“对外暴露的模型名”和“实际请求上游模型名”分离。

每条模型映射包含：

- `model`：对客户端暴露的模型名
- `target`：实际调用上游时使用的目标模型名

用途：

- 把客户端请求的模型名重定向到其他真实模型；
- 用统一别名屏蔽不同厂商模型命名差异；
- 为迁移旧客户端配置提供兼容层。

示例场景：

- 客户端请求 `gpt-4o`
- 网关内部将其映射到某个供应商的 `claude-3-7-sonnet`

模型映射生效位置：

- 命中路由规则后；
- 精确模型匹配后；
- 前缀推断后；
- 默认供应商或兜底供应商选择后。

---

### 3.6 路由规则

项目支持根据请求内容把流量路由到不同供应商或模型。

每条规则包含：

- `name`：规则名
- `matchType`：匹配方式
- `pattern`：匹配内容
- `providerId`：目标供应商
- `targetModel`：可选的强制目标模型
- `priority`：优先级
- `enabled`：是否启用

当前已实现的匹配方式：

- `model`：按模型名匹配，支持精确和 `*` 前缀通配
- `system_contains`：匹配 system 文本包含某内容
- `message_contains`：匹配任意消息包含某内容
- `user_contains`：匹配用户消息包含某内容
- `assistant_contains`：匹配助手消息包含某内容

规则执行特点：

- 按优先级从高到低排序；
- 只要命中第一条可用规则就立即返回；
- 若规则指定了 `targetModel`，则优先覆盖模型映射结果。

适用场景：

- 代码类问题走某个供应商；
- 翻译类问题走更便宜的供应商；
- 指定模型前缀流量强制走某上游；
- 将某些高成本请求导向特定模型。

---

### 3.7 路由决策顺序

当前请求进入网关后的供应商选择顺序如下：

1. **优先检查自定义路由规则**
2. **精确匹配模型名**：在供应商已知模型列表里找完全一致的模型
3. **按模型前缀推断供应商类型**
4. **使用网关默认供应商**
5. **兜底到第一个启用的供应商**

当前内置前缀推断包括：

- `gpt-` / `o1-` / `o3-` / `dall-e` / `text-` → `openai`
- `claude-` → `anthropic`
- `gemini-` / `models/` → `gemini`

这意味着即使没有显式规则，只要模型前缀可识别，也能自动选择合适供应商。

---

### 3.8 协议适配与转换

网关内部使用协议适配器机制处理不同上游 API。

当前从代码可以确认的适配器包括：

- `OpenAIAdapter`
- `OpenAIResponsesAdapter`
- `Anthropic` 适配器
- `Gemini` 适配器

系统行为分两类：

#### 3.8.1 透传模式

如果目标供应商本身就是 OpenAI 兼容接口：

- 网关只更新请求中的模型名；
- 然后直接把请求透传给上游；
- 返回值也直接透传。

#### 3.8.2 协议转换模式

如果目标供应商不是 OpenAI Chat 原生协议：

- 先把 OpenAI 风格请求转换成内部统一请求；
- 再由目标适配器构造原生请求；
- 接收上游返回后，再转回 OpenAI 风格响应；
- 流式响应也会按 SSE 形式转换。

这使客户端可以统一用 OpenAI SDK 接 Anthropic、Gemini 等上游。

---

### 3.9 流式响应支持

项目已实现流式请求识别与流式响应转发。

支持点：

- 自动识别 `"stream": true` 请求；
- 对 OpenAI 兼容上游直接透传流；
- 对 Gemini、Anthropic 等流式事件进行解析；
- 转换为 OpenAI 风格 SSE 输出；
- 输出 `[DONE]` 结束标识。

这意味着客户端可直接使用流式消费逻辑，无需感知不同厂商协议差异。

---

### 3.10 请求日志与审计

所有网关请求都会被写入本地数据库，用于排障和审计。

日志字段包括：

- 请求时间
- 请求方法与路径
- 请求模型 `model`
- 最终目标模型 `targetModel`
- 命中的供应商 ID / 名称 / 类型
- 端点模式
- 上游 Base 与 Path
- 是否流式
- HTTP 状态码
- 延迟毫秒数
- 错误信息
- 客户端 IP
- User-Agent
- 请求体摘要
- 响应头摘要
- 响应体摘要

日志功能支持：

- 按数量拉取最近日志；
- 在 UI 中查看明细；
- 筛选失败请求；
- 查看请求参数、上游目标与响应结果。

这是当前项目非常实用的一块功能，便于排查“为什么没命中预期供应商 / 为什么失败 / 为什么模型被映射”。

---

### 3.11 网关访问鉴权

网关支持配置一个本地访问密钥 `AuthKey`。

启用后，请求方需要满足以下任一方式：

- 请求头 `x-api-key: <key>`
- 请求头 `Authorization: Bearer <key>`

如果配置为空，则不做鉴权。

适用场景：

- 本机仅自用时可关闭；
- 局域网共享、脚本调用或多应用接入时可开启。

---

### 3.12 配置持久化

当前配置存储方式为本地 SQLite 数据库，数据库文件位于项目工作目录：

- `icoo_proxy.db`

兼容能力：

- 首次启动会自动初始化数据库结构；
- 如果数据库为空，会尝试从旧版 TOML 配置迁移；
- 旧版配置文件路径：`icoo_proxy.toml`

当前持久化的数据包括：

- 网关配置
- 供应商配置
- 路由规则
- 请求日志

从代码还可以确认存在密钥文件路径：

- `icoo_proxy.key`

说明项目预留了本地密钥材料存放能力。

## 9. 项目核心价值总结

`icoo_proxy` 的本质不是单纯的“供应商配置界面”，而是一个：

**面向本地桌面使用场景的多供应商 AI 协议统一网关。**

它提供的核心价值是：

- **统一入口**：客户端只对接一个 OpenAI 风格地址；
- **统一协议**：屏蔽不同厂商 API 差异；
- **统一路由**：按规则和模型决定流量去向；
- **统一观测**：通过日志和状态页做故障定位；
- **统一运维**：通过桌面端完成配置、测试和切换。
